# CLAUDE.md

## 项目概述

本项目是一个仅在本机运行的“秋招准备工作台”，用于把个人规划、每日/每周任务、岗位投递和 LeetCode 刷题记录集中管理。

核心目标：

- 用户可以手动创建任务，也可以上传规划文件或粘贴文本，由 AI 自动拆解任务。
- AI 生成的内容必须先进入可编辑草稿，只有用户确认后才成为正式任务。
- 用户可以查看今天和本周应该做什么，并直接点击完成。
- 用户可以维护目标岗位、岗位要求、投递记录、流程状态和下一步行动。
- 用户可以记录 LeetCode 刷题结果和复习计划。

## 首版功能范围

### 今日总览

- 展示今日、逾期和本周任务。
- 展示今日完成进度和本周完成进度。
- 展示岗位下一步行动和到期的 LeetCode 复习提醒。
- 支持直接完成或取消完成任务。

### 计划与任务

- 始终提供手动新建任务入口，且该功能不依赖 AI。
- 支持上传 `.md`、`.markdown`、`.txt`、`.docx` 和 `.pdf`。
- PDF 首版只支持可提取文字的文本型 PDF，不实现扫描图片 OCR。
- 不支持旧版二进制 `.doc`；界面应提示用户另存为 `.docx`。
- 支持直接粘贴计划文本。
- 上传或粘贴内容经过 AI 拆解后先生成任务草稿。
- 草稿可编辑、删除和补充，确认后批量写入正式任务。
- 正式任务支持新增、编辑、删除、筛选、完成和取消完成。

### 岗位投递

- 维护公司、岗位、地点、链接、岗位要求、投递日期、状态、下一步行动、下一步日期和备注。
- 状态为：目标岗位、准备投递、已投递、笔试、面试、Offer、拒绝。
- 下一步行动可以创建关联任务。
- 支持搜索、状态筛选和状态数量统计。

### LeetCode

- 维护题号、题名、链接、难度、标签、状态、解法摘要、完成日期和复习日期。
- 状态为：待刷、已完成、需复习。
- 到期复习记录应显示在今日总览。
- 支持按难度、标签和状态筛选，并展示本周完成数量和难度分布。

### 数据管理

- 所有业务数据保存在浏览器 IndexedDB。
- 支持完整 JSON 导出和恢复，备份必须包含版本号。
- 导入备份前先校验并展示摘要，用户确认后再整库替换。
- 恢复失败必须回滚，不能破坏现有数据。

## 技术架构

- 前端：Vite + React + TypeScript。
- 路由：React Router。
- 本地数据：IndexedDB，通过独立数据访问层使用，页面不得直接操作原生 IndexedDB API。
- 本地服务端：Node.js + TypeScript。
- AI：OpenAI 兼容接口，由本地服务端代理调用。
- 时区：`Asia/Shanghai`。
- 本周范围：周一至周日。

保持模块边界清晰：

- `Dashboard`：只做聚合与展示。
- `Tasks`：正式任务、任务草稿和任务查询。
- `Applications`：岗位与投递流程。
- `LeetCode`：题目、完成和复习记录。
- `Plan Import`：文件文字提取、AI 调用、结果校验与草稿生成。
- `Settings/Backup`：AI 配置状态、导出与恢复。

不要为了未来可能的需求引入微服务、云数据库、账号系统或通用插件框架。

## AI 接口与安全

使用以下环境变量：

```env
AI_BASE_URL=https://api.example.com/v1
AI_API_KEY=your-api-key
AI_MODEL=your-model-name
```

必须遵守：

- API Key 只能由本地服务端从 `.env` 读取。
- API Key 不得发送到前端、写入 IndexedDB、写入日志或包含在备份中。
- 提供 `.env.example`，禁止提交真实 `.env` 和密钥。
- 原始 DOCX/PDF 文件只在单次请求内处理，不写入磁盘或数据库，也不直接上传给 AI 供应商。
- 本地服务端只把提取后的计划文本发送给 AI。
- AI 返回结果必须做运行时结构校验，不能直接信任或写入数据库。
- AI 请求失败、超时或输出不合法时，不得修改正式任务。
- AI 未配置时，手动任务、岗位、LeetCode 和备份功能仍必须正常工作。

## 文件导入约束

- 文件上传端点只接受明确允许的扩展名和 MIME 类型。
- 单文件最大 20 MB。
- 提取后的文本最大 100,000 个字符，超限时明确拒绝，不静默截断。
- Markdown/TXT 按 UTF-8 读取。
- DOCX 提取标题、正文段落和表格文本，不承诺保留视觉排版。
- PDF 按页面顺序提取可选择文字。
- 加密、损坏、空文件或无法提取文字的扫描版 PDF 必须返回可理解的错误，并且不调用 AI。

## 数据与交互规则

- 正式任务来源只能是手动创建、计划导入或岗位行动。
- LeetCode 复习提醒使用派生查询，不复制为正式任务。
- AI 导入成功只生成草稿；确认草稿时使用单个 IndexedDB 事务创建正式任务、更新导入状态并删除草稿。
- 同一岗位、相同行动内容和日期已有未完成任务时，不重复创建。
- 岗位删除后保留已创建的关联任务，并显示来源已删除。
- 完成任务时记录 `completedAt`，取消完成时清空该字段。
- 日期使用 `YYYY-MM-DD`，时间戳使用 ISO 8601。
- 今日任务包含：计划日期为今天、截止日期为今天、以及已逾期且未完成的任务。

## 工程约束

- 只修改完成当前任务所需的文件，不重构无关代码。
- 优先复用项目已有模式；不要引入没有明确价值的依赖或抽象层。
- 领域逻辑与 React 组件分离，避免把数据库查询、日期规则或 AI 校验散落在页面组件中。
- 外部输入必须在边界处校验，包括表单、文件、AI 响应和备份 JSON。
- 错误消息要可操作，但不能泄露密钥、完整上游响应头或敏感配置。
- 不提交 `.DS_Store`、构建产物、依赖目录、真实 `.env` 或本地数据库数据。
- 新依赖必须说明必要性，并优先选择维护活跃、用途单一的库。

项目建立后应提供并维护以下脚本；具体命令以 `package.json` 为准：

- 开发启动
- 生产构建
- 单元与流程测试
- TypeScript 类型检查
- 代码检查

README 必须说明安装、AI 配置、本地启动、备份恢复和手动验收步骤。

## 测试要求

实现功能时优先先写失败测试，再写最小实现。至少覆盖：

- 今天、本周、逾期和跨周日期边界。
- 手动新建任务不调用 AI。
- Markdown、TXT、DOCX 和文本型 PDF 的成功提取。
- 损坏、加密、超限文件和扫描版 PDF 的失败路径。
- AI 输出结构校验与非法响应拒绝。
- 草稿确认事务成功与回滚。
- 岗位行动任务的关联和去重。
- LeetCode 到期复习提醒。
- 任务完成后总览进度更新。
- JSON 备份格式校验、恢复成功与恢复回滚。
- 页面刷新后的 IndexedDB 数据持久化。

不要为了小改动新建另一套测试框架；使用项目已有测试工具。

## 完成前验证

在声称任务完成前：

1. 运行与改动相关的测试、类型检查、代码检查和构建。
2. 检查测试是否真正覆盖本次需求，而不只是命令返回成功。
3. 对无法自动验证的 UI 或真实 AI 接口给出明确的手动验收步骤。
4. 汇报修改文件、实际执行的命令、结果和仍存在的限制。
5. 不以“没有发现错误”代替需求逐项验证。
